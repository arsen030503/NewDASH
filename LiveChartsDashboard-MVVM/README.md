# LiveChartsDashboard — refactored to real MVVM

## What changed vs. the original repo

| Area | Before | After |
|---|---|---|
| Architecture | Code-behind (`MainWindow.axaml.cs` sets `DataContext = this`, holds all logic) | Real MVVM: `ViewModels/MainWindowViewModel.cs` holds all state/logic, window is just a shell |
| Change notification | None (`DataContext = this`, no `INotifyPropertyChanged`) | `ObservableObject` + `[ObservableProperty]` (CommunityToolkit.Mvvm source generators) |
| Button → action | `Click="GetWeatherButton_Click"` event handler in code-behind | `Command="{Binding GetWeatherCommand}"` — generated by `[RelayCommand]` |
| Data binding | Controls referenced by `x:Name` and set directly (`CityName.Text = ...`) | XAML `{Binding ...}` to ViewModel properties, `x:DataType` for compiled bindings |
| Database | `DatabaseService` used `Microsoft.Data.SqlClient` (SQL Server driver) with a **Postgres-style connection string** pointed at a DB that was never created — while `DatabaseInitializer` was actually creating a **separate SQLite file**. Two unrelated data stores, only one of which was real. | `DatabaseService` now consistently uses `Microsoft.Data.Sqlite` against the same `database.db` that `DatabaseInitializer` creates. One real, working path. |
| `Activity` model | Defined twice (once in `Models/Activity.cs`, once again inside `DatabaseService.cs`) | Single definition in `Models/Activity.cs` |
| `DatabaseInitializer` | Defined twice (standalone file **and** nested inside `Program.cs`) | Single definition |
| Packages | `Microsoft.Data.SqlClient`, `Npgsql`, `Npgsql.EntityFrameworkCore.PostgreSQL`, `Microsoft.EntityFrameworkCore` referenced but not meaningfully used | Removed the unused/conflicting ones, added `CommunityToolkit.Mvvm` |

## What's still honestly true about this project (be ready to say so)

- There's still no real-time push (no timer/socket) — weather is fetched on button click, activity chart data loads once on startup. If asked "is this actually real-time?", the honest answer is "no, it's on-demand / one-shot loads, not a live stream."
- The "database" story is now internally consistent (SQLite only) rather than being framed as "SQLite + PostgreSQL working together" — because in the original code PostgreSQL was never actually reachable.

## Building & testing this yourself

This sandbox can't reach `nuget.org`, so I wasn't able to `dotnet restore`/`build` this to confirm it compiles cleanly. Please do that locally before the interview:

```bash
cd LiveChartsDashboard
dotnet restore
dotnet build
dotnet run
```

Things to double check locally:
- `CommunityToolkit.Mvvm` source generators require the project to target a recent SDK (net8/net9 is fine here) — if you see errors about `[ObservableProperty]`/`[RelayCommand]` not generating members, do a clean rebuild (`dotnet clean && dotnet build`).
- `x:DataType="vm:MainWindowViewModel"` enables Avalonia's *compiled* bindings — if you rename a ViewModel property, XAML binding errors will show up at build time instead of silently failing at runtime, which is one of the real advantages of doing this properly.
